# 服务器版行程编辑器设计

> 此方案已被纯 HTML + localStorage 方案取代，保留仅供参考。当前设计见 `2026-07-16-static-itinerary-editor-design.md`。

## 目标

将现有“武汉 → 山东 9 日旅行计划”静态页面改为可部署的个人行程编辑器。页面需要：

- 按当天当前顺序显示固定像素大小的数字标记。
- 路线按数字顺序连接并显示方向箭头。
- 切换 DAY 标签时立即显示当天全部标记和完整路线。
- 支持地图选点或地点搜索后新增行程点。
- 支持拖拽当天行程卡片排序。
- 所有修改自动保存到服务器，任何设备再次打开时均读取最新结果。

系统只有一份共享行程，不设置账号、密码或多用户隔离。

## 总体架构

采用同源的单体部署：

- 浏览器前端：保留当前单页界面和腾讯地图 JSAPI GL。
- Python FastAPI：提供行程、排序和地点搜索 API，同时托管静态页面。
- SQLite：作为行程的唯一持久化数据源。
- Docker：封装运行环境并挂载 SQLite 数据卷。
- Nginx：可选，用于 HTTPS、域名和反向代理。

前端不再把内嵌 `POIS` 作为运行时真相。打开页面时通过 API 读取数据库；每次新增或排序成功后，再按后端返回的数据重画时间轴、编号和路线。

## 数据模型

`points` 表包含：

- `id`：稳定 UUID。
- `day`：第几天，范围 1–9。
- `position`：当天自定义顺序，从 1 开始且不重复。
- `time`：`HH:MM`。
- `title`、`name`、`description`、`city`。
- `category`：`scenic`、`food`、`culture`、`shop`、`transit` 或 `hotel`。
- `latitude`、`longitude`。
- `source`：`seed`、`map_click` 或 `place_search`。
- `created_at`、`updated_at`。

服务器首次启动且数据库为空时，从 `coords.json` 导入基础行程。以后重启或重新部署不会重复导入，也不会覆盖用户修改。

## API

### 读取行程

`GET /api/itinerary`

返回所有点，先按 `day`、再按 `position` 排序。前端初次加载及保存失败后的恢复均以此结果为准。

### 新增行程点

`POST /api/points`

请求包含日期、时间、标题、描述、城市、类别、经纬度和来源。服务器校验后，默认按时间插入当天顺序；相同时间追加到已有点之后，并在一个事务中重排 `position`。

### 调整当天顺序

`PUT /api/days/{day}/order`

请求体为当天全部点 ID 的有序数组。服务器验证 ID 无缺失、无重复且都属于该天，然后在一个事务中更新顺序。验证失败时不写入任何部分结果。

### 搜索地点

`GET /api/places/search?keyword=...&city=...`

后端调用腾讯地图 WebService，限制结果数量并返回名称、地址、城市和坐标。WebService Key 由服务器环境变量提供，不发送给浏览器。

腾讯地图 JSAPI Key 必须出现在浏览器脚本地址中，无法作为秘密隐藏；部署时应在腾讯位置服务控制台配置允许访问的域名。

## 前端交互

### 标签切换和地图绘制

点击 DAY 标签后按以下固定流程执行：

1. 从内存数据中取出该天点位并按 `position` 排序。
2. 重建左侧卡片及从 1 开始的编号。
3. 为每个点生成带编号和类别颜色的 SVG 标记。
4. 清空旧图层，绘制当天所有标记。
5. 按编号顺序绘制一条 `MultiPolyline`，设置 `showArrow: true`。
6. 使用 `fitBounds` 将当天所有点放入当前视野。

标记保持固定屏幕像素大小，不聚合、不因缩放主动隐藏。地图缩得过小时，位置接近的编号允许相互遮挡。

### 新增点

“添加行程点”打开侧边表单，默认选择当前 DAY。位置可通过两种方式确定：

- 地图选点：进入选点状态，点击地图取得经纬度。
- 地点搜索：输入关键词和城市，从服务器返回的候选项中选择。

日期、时间、标题、类别和坐标为必填项。提交期间禁用按钮；成功后切换或保持目标 DAY，重新载入并绘制数据；失败时保留表单内容供重试。

### 拖拽排序

左侧当天卡片使用原生拖拽交互。落点后先更新界面编号和路线，再提交完整 ID 顺序：

- 成功：以后端顺序覆盖本地内存状态。
- 失败：恢复拖拽前顺序并显示错误提示。

拖拽排序后，编号以用户顺序为准，不再强制按时间重新排序；新增点仍按时间确定首次插入位置。

## 一致性与错误处理

- SQLite 写操作使用事务，避免只保存部分顺序。
- API 对日期、时间、类别、坐标范围和点 ID 做服务端校验。
- 页面加载失败时显示重试入口，不使用过期静态数据假装保存成功。
- 地点搜索失败时仍允许地图点击选点。
- 保存过程中阻止重复提交和并发拖拽。
- 前端只在 API 成功后宣告保存完成。

## 部署与数据安全

- SQLite 文件保存到 Docker 持久化卷，升级容器时不得删除该卷。
- 提供数据库文件备份说明；备份该文件即可恢复全部行程。
- API 与页面同域，CORS 不开放给其他来源。
- 因明确不设置认证，任何能够访问网站的人都能修改行程。若服务器直接暴露到公网，应通过防火墙、VPN 或 IP 白名单限制访问范围。

## 验收与测试

- 数据库首次启动准确导入 `coords.json`，二次启动不重复导入。
- 每个 DAY 切换后只显示当天点位，编号从 1 连续递增，路线包含全部点并带箭头。
- 放大和缩小地图时编号标记保持固定尺寸且不被聚合。
- 地图点击和地点搜索均能新增点，刷新及更换设备后仍存在。
- 拖拽后编号、卡片、路线同步变化，刷新后顺序保持。
- 非法坐标、缺失字段、重复或缺失排序 ID 不会写入数据库。
- 模拟 API 失败时，新增表单保留数据，拖拽顺序能够回滚。
- Docker 容器重建后，挂载卷中的行程仍可读取。
